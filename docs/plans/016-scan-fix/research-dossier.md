## Executive Summary

### What's Happening
GDScript (`.gd`) files are NOT being indexed by fs2 because the `.gd` file extension is **not mapped** in the `EXTENSION_TO_LANGUAGE` dictionary, even though tree-sitter-language-pack has a working `gdscript` grammar available.

### Business Impact
- Users scanning Godot game projects get no GDScript code indexed
- Semantic search, code navigation, and smart content generation are unavailable for .gd files
- The dig-game repo has 347 .gd files that are completely invisible to fs2

### Root Cause
Missing 2-line configuration in `src/fs2/core/adapters/ast_parser_impl.py`:
1. `.gd` extension not in `EXTENSION_TO_LANGUAGE` dict
2. `gdscript` not in `CODE_LANGUAGES` set

### Quick Stats
- **dig-game graph**: 1,485 nodes indexed (C#, markdown, JSON, etc.)
- **GDScript files in repo**: 347
- **GDScript files in graph**: 0
- **Fix complexity**: 2 lines of code

## Architecture Context

### Language Detection Flow

```
file_path → detect_language() → EXTENSION_TO_LANGUAGE lookup
                                        ↓
                              Found? → language string → parse()
                              Not found? → None → skip file (warning logged)
```

**Code path** (`ast_parser_impl.py:220-249`):
```python
def detect_language(self, file_path: Path) -> str | None:
    # Check exact filename match (Dockerfile, etc.)
    # Check extension
    suffix = file_path.suffix.lower()
    if suffix in EXTENSION_TO_LANGUAGE:
        return EXTENSION_TO_LANGUAGE[suffix]
    return None  # ← .gd files hit this path
```

### Node Classification Flow

```
language → CODE_LANGUAGES check
                ↓
        In set? → ContentType.CODE → extract callables/types
        Not in set? → ContentType.CONTENT → file-only node
```
