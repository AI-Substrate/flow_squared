IC-01: GraphStore ABC – Primary Data Source Interface
────────────────────────────────────────────────────────────────────────────
File: src/fs2/core/repos/graph_store.py
Type: Abstract Base Class (READ-ONLY for reports)

CRITICAL METHODS:
  ✓ get_all_nodes() → list[CodeNode]                 # Full inventory
  ✓ get_node(node_id: str) → CodeNode | None         # Single fetch
  ✓ get_edges(node_id, direction, edge_type) → list  # Cross-file refs
  ✓ get_children(node_id) → list[CodeNode]           # Tree structure
  ✓ get_parent(node_id) → CodeNode | None            # Hierarchy
  ✓ get_metadata() → dict                            # Scan metadata

EDGE SEMANTICS:
  • Containment: parent → child (implicit hierarchy via edges)
  • References: edge_type="references" (cross-file dependencies)
  • Query pattern: get_edges(..., direction="outgoing|incoming|both", edge_type=None)

CONTRACT:
  ✓ Node ID format: {category}:{file_path}:{qualified_name}
  ✓ Metadata keys: format_version, created_at, node_count, edge_count
  ✓ Reports are read-only (never call add_node/add_edge/save/clear)
  ✓ Lazy loading: graph loads on first service access


IC-02: CodeNode Model – Universal Node Schema (23 Fields)
────────────────────────────────────────────────────────────────────────────
File: src/fs2/core/models/code_node.py
Type: Frozen Dataclass (@dataclass(frozen=True))

FOR VISUALIZATION:
  node_id          str            Unique ID; split to extract file_path
  name             str|None       Display name; use qualified_name as fallback
  category         str            Universal: "file", "callable", "type", etc.
  qualified_name   str            Hierarchical name (e.g., "Calculator.add")
  start_line       int            1-indexed; human-readable line number
  end_line         int            Span end line for UI highlighting
  content          str            Full source code (embed in detailed reports)
  signature        str|None       Declaration first line(s) for quick preview
  language         str            Source grammar (Python, JavaScript, etc.)
  smart_content    str|None       AI-generated summary (use in smart reports)

NEVER INCLUDE IN REPORTS (Security):
  ✗ embedding, smart_content_embedding        Vectors → data leak risk
  ✗ content_hash, smart_content_hash          Internal change detection
  ✗ start_byte, end_byte, start_column, etc.  Tree-sitter internals
  ✗ is_error, truncated, field_name, etc.    Implementation details

DESIGN PRINCIPLES:
  • Language-agnostic: Works for any tree-sitter grammar
  • Dual classification: ts_kind (grammar-specific) + category (universal)
  • No embedded children: Hierarchy via graph edges, not nested objects
  • Frozen: Immutable after creation

IC-08: Cross-File Relationships – Reference Edge Semantics
────────────────────────────────────────────────────────────────────────────
Files: src/fs2/core/repos/graph_store.py + src/fs2/mcp/server.py

REFERENCE QUERY PATTERN:
  # Get nodes that reference this node
  incoming = store.get_edges(
    node_id="callable:src/calc.py:Calc.add",
    direction="incoming",
    edge_type="references"
  )
  # Returns: [("callable:src/main.py:main", {"edge_type": "references"}), ...]

  # Get nodes this node references
  outgoing = store.get_edges(
    node_id="callable:src/calc.py:Calc.add",
    direction="outgoing",
    edge_type="references"
  )

EDGE TYPES:
  • Containment (no edge_type kwarg):     Parent → child hierarchy
  • References (edge_type="references"):  Cross-file dependencies

REPORTS USE CASE:
  "Dependencies Report": Show which nodes reference a given node (used-by)
  "Impact Report":      Show which nodes a given node references (uses)

================================================================================
DESIGN PRINCIPLES FOR REPORTS IMPLEMENTATION
================================================================================

1. DEPENDENCY INJECTION
   Reports service receives ConfigurationService + GraphStore (ABC)
   Never depend on implementations (NetworkXGraphStore, etc.)

2. LAZY LOADING
   Services auto-load graph on first access
   Reports don't manage graph lifecycle or persistence

3. READ-ONLY
   Reports NEVER call: add_node, add_edge, save, clear, set_metadata
   Graph is immutable from reports perspective

4. NO CACHING
   Per R3.5: All graph access through GraphStore on-demand
   Enables real-time updates without cache invalidation

5. SECURITY
   Use _code_node_to_dict for JSON output
   Prevents embedding/hash/internal field leakage

6. THREAD-SAFE
   Services use lazy loading + RLock internally
   Safe for background report generation

7. LANGUAGE-AGNOSTIC
   Filter by category (universal), not ts_kind (grammar-specific)
   Reports work uniformly across Python, JS, Go, Rust, etc.

8. ERROR HANDLING
   Catch: GraphNotFoundError, GraphStoreError, MissingConfigurationError
   Degrade gracefully in reports UI
